## Refined Findings

### 1. Web id drift is semantic, not only prefix drift

Observed meanings:

- daemon web sensor:
  - `WEB-05` = default credentials
  - `WEB-08` = CORS wildcard
  - `WEB-09` = missing security headers
  - `WEB-10` = technology identified
- active collector:
  - `WB-05` = sensitive paths exposed
  - `WB-08` = default credentials
  - `WB-19` = missing security headers
- checked-in catalog:
  - `WB-05` = admin interface exposed
  - `WB-08` = git exposure
  - `WB-19` = Elasticsearch unauth
  - `WB-20` = Jenkins script console

Conclusion:

The active collector, daemon web sensor, and checked-in catalog are not using one shared web vocabulary.

### 2. The semantic collision already produces false positive projections

Synthetic validation:

- emitted `WB-01=realized`
- emitted collector-style `WB-05=realized` with detail `Sensitive paths accessible`
- projected against checked-in `web_initial_access_v1`
- result: `classification=realized`, because the checked-in path requires `WB-01` and `WB-05`

Conclusion:

This is not merely stale naming. Current collector semantics can satisfy a checked-in path for the wrong reason.

### 3. Daemon web sensor events do not enter the checked-in web projector

Synthetic validation:

- built `WEB-01` and `WEB-05` events via `skg.sensors.web_sensor._ev()`
- projected with toolchain `skg-web-toolchain` and path `web_surface_v1`
- result: no interpretation output

Conclusion:

The daemon-native web sensor is currently outside the checked-in web projection path.

### 4. Workload identity shape is unstable across equivalent operator flows

- daemon `/collect` defaults `workload_id` to raw target string
- daemon-offline CLI SSH fallback uses `ssh::{ip}`
- CLI web observe omits `--workload-id`
- active collector then defaults to bare hostname/IP
- gravity and structured fetch typically use `web::{ip}`
- tests overwhelmingly assume prefixed forms like `host::...`, `ssh::...`, and `web::...`

`parse_workload_ref()` confirmation:

- raw `10.0.0.9` and `ssh::10.0.0.9` share `identity_key=10.0.0.9`
- but produce different `manifestation_key` values

Conclusion:

Equivalent operator flows can fragment one identity into multiple manifestation tracks.
