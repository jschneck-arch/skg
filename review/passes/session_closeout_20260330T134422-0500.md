## Refined Findings

### 1. `report` is only partially offline and can silently present a degraded view

- `cmd_report()` reads surface state locally
- folds come from `_api("GET", "/folds") or {"folds": []}`
- substrate self-audit also depends on daemon endpoints for resonance, feedback, graph, folds, and ollama status
- `_api()` returns `None` on daemon `URLError` without surfacing a warning in `report`

Conclusion:

`skg report` can look valid while omitting folds and other daemon-backed state when the daemon is not running.

### 2. `cmd_proposals` is riskier and more drifted than its surface suggests

- there are no direct CLI tests for `cmd_proposals`
- the module docstring says "direct file read, no import dependencies", but `trigger`, `accept`, `reject`, and `defer` all import live runtime modules
- the trigger path still contains generic-shell DVWA/CMDI-specific exploit delivery logic

Conclusion:

The operator proposal shell is not a thin queue UI; it is an active execution surface with weak direct coverage and some stale self-description.

### 3. Post-session host projection is hardcoded to one path that is too narrow

- `_run_post_projection()` always invokes the host projector with `host_linux_privesc_sudo_v1`
- that helper is called whenever a Metasploit session is opened
- the host projector itself defaults to `host_ssh_initial_access_v1`
- the host catalog defines multiple host paths, including SSH initial access, WinRM initial access, sudo privesc, SUID privesc, kernel privesc, post-exploitation, and network exploit

Conclusion:

Session-backed proposal execution currently collapses diverse host outcomes through one Linux sudo-privesc lens.

### 4. `proposals list` contains a dead branch for toolchain maturity display

- the list view truncates `proposal_kind` to `[:14]`
- it then compares the truncated value to `"toolchain_gener"`
- shell validation confirmed `"toolchain_generation"[:14] == "toolchain_gene"`

Conclusion:

Toolchain proposal maturity annotations are currently unreachable in the list view.

### 5. CLI coverage remains shallow on operator-facing reporting/proposal flows

- `cmd_engage` only has a missing-subcommand usage test
- no direct tests were found for `cmd_report`, `cmd_calibrate`, or `cmd_proposals`
- `tests/test_cli_commands.py` still leans heavily on importability/callability checks for `skg.cli.utils`
- `tests/test_bwapp_data.py` provides meaningful data-domain coverage, but it does not validate the report/proposal operator layer

Conclusion:

The CLI shell is less directly validated than the substrate and several domain/toolchain components it wraps.
