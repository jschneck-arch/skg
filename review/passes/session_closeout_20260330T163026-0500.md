## Refined Findings

### 1. `db_discovery` is operator-facing, but not on the canonical event contract

- `skg data discover` calls `run_with_profiling()`
- DE-* events emitted by `db_discovery` omit both `provenance` and `attack_path_id`
- DP-* profiling is better integrated than the DE-* security-surface side

Conclusion:

The data discovery bridge is real, but its security-surface events are still on a sidecar envelope.

### 2. The IoT firmware probe has a concrete HTTP-only false-negative path

- `probe_device()` decides reachability from spontaneous TCP banners
- it returns before any HTTP request when banners are absent
- `probe_network_only()` exists but is not wired into the main path

Conclusion:

Ordinary HTTP-admin devices can be missed by the main IoT probe.

### 3. The stale root web projector is actively unreliable

- it points at a non-existent catalog file
- its `latest()` helper ignores the provided `workload_id`

Conclusion:

The root web projector should not be treated as a supported alternate path without repair.

### 4. Coverage on this batch is uneven

- `db_discovery` has unit-style coverage in `test_bwapp_data.py`
- the opened AI/IoT/nginx/metacognition/root-web helpers have little or no direct coverage

Conclusion:

These toolchain-side helpers remain a weakly pinned part of the repo compared with the substrate core.
