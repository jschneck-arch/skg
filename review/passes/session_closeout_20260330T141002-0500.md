## Refined Findings

### 1. The operator shell still does not share one authority boundary

- `report` and `surface` partially merge measured view state
- `status` and `web` still rank raw discovery surfaces
- fold helpers choose between daemon and disk state by row count rather than freshness

Conclusion:

The CLI shell still presents multiple competing authority models depending on which command is used.

### 2. `cmd_surface` projection output is not a clean per-target view

- it loads interp artifacts from `SKG_STATE_DIR/interp`, `DISCOVERY_DIR`, and `/tmp`
- it keeps only one "best" row per `attack_path_id`
- validation with two targets and two interp files sharing `web_initial_access_v1` produced only one displayed projection row

Conclusion:

The generic surface projection summary can hide target-specific disagreements and leak stale loose artifacts into operator output.

### 3. `cmd_replay` is approximate, not live-equivalent

- the command claims equivalence to the live gravity cycle
- it imports support/state engines
- it then ignores them and collapses wickets with a local majority-vote routine

Conclusion:

`skg replay` should currently be treated as a convenience inspection path, not a faithful substrate replay.

### 4. `cmd_check` has a concrete dependency probe bug

- it validates WinRM via `__import__("pywinrm")`
- the runtime imports `winrm`
- validation in this environment confirmed `winrm` imports while `pywinrm` does not

Conclusion:

`skg check` can falsely warn that WinRM support is missing when it is actually installed.

### 5. The dedicated MSF-session adapter still emits non-canonical workload ids

- session-backed host events use raw host strings like `10.0.0.7`
- the adapter default workload is `msf_workspace`
- offline execution confirmed raw workload ids in emitted `HO-17` / `HO-10` events
- the CLI proposal-trigger shell instead rewrites post-session evidence to `host::...`

Conclusion:

MSF-derived host evidence still fragments across multiple workload-id shapes.

### 6. Coverage remains shallow on these operator-shell paths

- no direct tests were found for `cmd_surface`, `cmd_web_view`, `cmd_status`, `cmd_start`, `cmd_stop`, `cmd_mode`, `cmd_identity`, `cmd_field`, or `cmd_gravity`
- `cmd_check` and `cmd_replay` do have direct tests, but they are shallow shell-level checks
- no direct tests were found for `skg-host-toolchain/adapters/msf_session/parse.py`

Conclusion:

This layer remains substantially less validated than the core substrate and several gravity/library helpers.
