## Refined Findings

### 1. Resonance drafting is internally shape-inconsistent, not just CLI-drifted

- `ResonanceEngine.surface()` returns `{"record": ..., "score": ...}` dictionaries
- `_build_user_prompt()` iterates those buckets as if they were `(record, score)` tuples
- direct validation confirmed `draft_prompt()` raises `TypeError: string indices must be integers, not 'str'` against the current engine-style payload

Conclusion:

The resonance drafting mismatch is not limited to missing CLI exposure; the helper/engine contract is currently broken.

### 2. The standalone resonance CLI has diverged from the engine and the main CLI

- `skg/resonance/cli.py` implements `drafts` by calling `engine.list_drafts()`
- `ResonanceEngine` has no `list_drafts()` method
- the top-level `skg` CLI avoids this only because it lists drafts by scanning files directly

Conclusion:

There are now two resonance command surfaces with different behavior, and the standalone one is internally broken.

### 3. Resonance ingester completeness is overstated

- adapter discovery is driven by hard-coded evidence-source tables and regex scanning, not full structured adapter metadata
- `DomainMemory.adapters` is modeled but not populated during ingest
- targeted validation confirmed ingest summaries count processed domains as if they were newly added

Conclusion:

Resonance memory is real, but parts of its advertised domain/adapter linkage remain ceremonial or incomplete.

### 4. CLI coverage is still broad-but-shallow

- full read of `tests/test_cli_commands.py` confirms many command surfaces only have smoke/help tests
- no direct behavioral tests were found for `cmd_report`, `cmd_calibrate`, `cmd_resonance`, `cmd_derived`, `cmd_aprs`, `cmd_escape`, `cmd_lateral`, or `cmd_catalog`

Conclusion:

The operator shell remains less tightly pinned by tests than the substrate and helper layers beneath it.
