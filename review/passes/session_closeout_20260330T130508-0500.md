## Refined Findings

### 1. `build_engagement_db()` misses transitions under default runtime layout

- `engagement_dataset.py` defaults `delta_dir` to `SKG_STATE_DIR`
- canonical path constants place transitions under `DELTA_DIR = SKG_STATE_DIR / "delta"`
- isolated validation confirmed: default build ingested `0` transitions, explicit `delta_dir=DELTA_DIR` ingested `1`

Conclusion:

The engagement dataset currently drops transition history by default unless the caller passes `delta_dir` explicitly.

### 2. Engagement referential integrity is keyed on manifestation strings, not stable identity

- observations store both `workload_id` and `node_key`
- projections store only `workload_id`
- DP-05 uses exact `workload_id` membership
- `engage clean` deletes orphaned projections with the same exact-string join
- earlier passes already confirmed raw and prefixed workload ids can represent the same identity

Conclusion:

The engagement-analysis path is structurally vulnerable to false orphaning and cleanup loss when workload-id drift is present.

### 3. Missing CLI surfaces are confirmed runtime behavior, not just doc drift

- live parser inventory shows no top-level `train` command
- live parser inventory shows `data` subcommands limited to `catalog`, `discover`, `paths`, `profile`, and `project`
- module docs still advertise `skg train run` and `skg data redteam`

Conclusion:

Training and cross-domain data-derivation capabilities exist in code but are not exposed through the advertised CLI surface.

### 4. Proposal corpus duplication remains the strongest confirmed training-data defect

- `accept()` calls `on_proposal_accept()` twice
- `reject()` calls `on_proposal_reject()` twice
- `training.corpus` appends one new shard per call with no dedupe

Conclusion:

Proposal review outcomes are likely double-counted in the training corpus.

### 5. Coverage remains shallow on newer learning/engagement paths

- direct tests were not found for `skg.training.corpus`, `skg.training.scheduler`, or `skg.training.trainer`
- `engagement_dataset` coverage still appears limited to CLI mocking

Conclusion:

These layers are live but not validated to the same standard as core substrate and sensor/projection paths.
