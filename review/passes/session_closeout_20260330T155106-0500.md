## Refined Findings

### 1. `derived rebuild` is not currently a pure substrate reconstruction

- the command/help text says derived state is rebuilt from append-only substrate
- the implementation reprojects both `SKG_STATE_DIR/events` and `DISCOVERY_DIR`
- fold rebuild also depends on the current latest surface and fold-location string matching

Conclusion:

`skg derived rebuild` is currently a hybrid reconstruction path, not a clean substrate-only rebuild.

### 2. Dark-hypothesis planning is implemented, but its runtime integration is partial

- instrument discovery only scans `SKG_STATE_DIR/toolchains`
- built-in checked-in toolchains are invisible to that helper
- the sensor writes `cognitive_action` proposals directly into `state/proposals`
- the shared proposal trigger path only accepts `field_action`, and generic accept expects `staged_path` for unknown kinds

Conclusion:

The dark-hypothesis planner is a sidecar feature today: it can draft proposal files, but it is not fully integrated with the canonical toolchain inventory or the shared proposal lifecycle.

### 3. The `field` CLI domain enum is stale

- the parser allows only a narrow hardcoded domain list
- active `binary` and `data_pipeline`-style values are rejected at parse time
- runtime regressions confirm binary identity/projection paths are active elsewhere in the repo

Conclusion:

The `field` command lags the live domain inventory and currently blocks some real workloads from the CLI surface.

### 4. Coverage is uneven across this remaining CLI/support layer

- `cmd_graph`, `cmd_feed`, and `cmd_core` have direct but shallow command tests
- no direct tests were found for `cmd_resonance`, `cmd_derived`, `cmd_aprs`, `cmd_escape`, `cmd_lateral`, or `cmd_catalog`
- `test_dark_hypothesis_sensor.py` only covers prompt/JSON planning behavior and patches away real toolchain discovery
- `test_runtime_regressions.py` is stronger than expected and directly covers assistant custody, binary aliasing, projector discovery, observed-tool overlays, and dynamic sensor discovery

Conclusion:

This layer is partly real and partly under-validated; the strongest gaps are around lifecycle integration and CLI/domain exposure, not the existence of the modules themselves.
