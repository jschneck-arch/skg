## New Concrete Findings

1. The core substrate model is implemented, not ceremonial.
   - Identity canonicalization, support aggregation, tri-state collapse, path projection, feedback ingestion, graph priors, and pearl-manifold memory all exist as real code paths with coherent contracts.

2. The assistant boundary is one of the stricter parts of the repo.
   - `assistant_contract.py` explicitly classifies outputs.
   - Non-observation assistant outputs are inadmissible to the observation plane.
   - Contract-backed artifacts are written with metadata and then wrapped into reviewable proposals.
   - Focused tests for that flow pass.

3. Proposal lifecycle is real and persistent.
   - `forge.proposals` persists pending/accepted/rejected/deferred/superseded records.
   - Triggered field-action proposals can later execute and feed new event artifacts back into runtime state.
   - This aligns with the operator-gated observation loop described in later docs.

4. The operator shell remains more scenario-specific than the docs imply.
   - `skg/cli/commands/proposals.py` contains hardcoded DVWA/CMDI payload delivery logic in a generic proposal execution helper.
   - This is useful for lab workflows, but it is not a clean substrate boundary.

5. `install.sh` is now clearly stale, not just generally old.
   - Its substrate smoke assertion for `BondState.prior_influence` no longer matches live code.

6. `skg.intel.surface` has contract drift.
   - No `SurfaceBuilder` exists.
   - The module docstring overstates what the live `surface()` path consumes.
   - `delta_store` remains in the function signature but is not used by `surface()` itself.

7. The daemon contains at least one concrete API hygiene defect.
   - `/delta/workloads` is defined twice with identical decorators and bodies.
