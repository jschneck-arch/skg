  Critical Breaks (Numbered from your own review docs)

  1. skg target remove is a stub (bin/skg:1779-1782)
  Two print statements, zero logic. Deleted targets persist in pearls.jsonl with full
  entropy scores and continue driving gravity cycles. This is the most disruptive
  operational bug — gravity wastes cycles scanning nonexistent hosts indefinitely.

  2. MSF session detection is broken cross-process (gravity_field.py:1658-1671)
  _exec_post_exploitation spawns a new msfconsole subprocess to check for sessions. The
  exploit that was triggered runs in a completely separate msfconsole process. Two
  independent processes have disjoint session spaces. msf_sensor.py already has
  _drain_sessions() using pymetasploit3 RPC — the correct path — but it's never called
  here.

  3. Windows post-exploitation sends Linux bash commands (gravity_field.py:1637-1656)
  The post-exploitation command string is pure bash: whoami && id, sudo -l, find / -perm
   -4000, cat /etc/passwd, mysql -u root. None of these exist on Windows. The dc01
  engagement confirmed every post-exploitation step was manually bridged.

  4. Credential harvest never writes to CredentialStore (gravity_field.py → exploit
  chain)
  _exec_post_exploitation parses /etc/shadow and env secrets but never calls
  CredentialStore().add(...). The cred_reuse instrument is correctly wired in the
  gravity loop but is starved — it can only reuse credentials manually placed there.

  5. MSF sensor → wrong wicket domain (msf_sensor.py:404)
  When a Meterpreter session opens, the sensor emits CE-01 ("container privileged" — a
  container escape wicket). A session on a bare-metal/VM target should realize HO-10
  (elevated privileges) and HO-05/HO-03 (initial access). Every session gets filed as a
  container escape precondition; host_winrm_initial_access_v1 never gets its wickets
  confirmed through the session sensor.

  6. Engage report deduplication missing (engagement_dataset.py:807-811)
  SELECT ... FROM projections WHERE classification='realized' has no GROUP BY
  (workload_id, attack_path_id). Projection files without an id field get a fresh uuid4
  at ingest, so the same realized path from two runs gets two rows. Reports show N
  copies of the same path per target.

  High Gaps (Not Bugs — Missing Capabilities)

  7. Async exploit execution spawns and forgets (gravity_field.py:5419-5434)
  Sync auxiliary proposals feed evidence back automatically (output → wicket events).
  Async exploit proposals — the ones that actually matter — get Popen'd with
  start_new_session=True and nothing else. No session polling, no post-exploitation, no
  wicket events. The only evidentially-complete path is auxiliary observation-class
  proposals.

  8. No Windows toolchain
  skg-host-toolchain is SSH-centric. DEFAULT_ATTACK_PATH is host_ssh_initial_access_v1.
  skg collect --target <ip> --method winrm hits the timeout (now fixed to 120s) but
  produces no output because there's no WinRM adapter that maps commands to wicket
  events. Everything Windows required manual NDJSON authoring.

  9. EXPLOIT_MAP missing Windows lateral paths (exploit_dispatch.py)
  Present: host_winrm_initial_access_v1, host_network_exploit_v1 (EternalBlue).
  Missing: RDP, SMB/PsExec, WMI, Pass-the-Hash, token impersonation, DCSync.
  ad_dcsync_v1 exists in the catalog but requires SESSION: {session_id} with no
  mechanism to bind a session ID automatically.

  10. Kuramoto graph propagation never called (skg/graph/__init__.py)
  skg/graph/__init__.py has a complete, mathematically correct Kuramoto prior
  propagation implementation. It is never imported in gravity_field.py. The network-wide
   intelligence that Paper 4 describes — priors elevating on neighboring targets after a
   host is compromised — is completely inactive. Cross-target contagion runs nowhere.

  11. LLM pool wired only to catalog drafting (gravity_field.py →
  _get_resonance_engine())
  _get_resonance_engine() is called only from _create_toolchain_proposals_from_folds() —
   for generating new catalogs from fold observations. It is never called to interpret
  realized paths, prioritize competing proposals, parse ambiguous MSF output, or
  generate engagement narrative. The llm_pool.py multi-backend infrastructure
  (Ollama/Anthropic/HuggingFace) exists and runs; it just isn't connected to any
  reasoning decision point.

  12. skg engage clean / skg engage fix missing
  DP-03 (NULL workload_id), DP-04 (evidence_rank out of bounds), DP-05 (orphaned
  projections) are detected but there's no repair command. Violations accumulate across
  engagements with no recovery path.

  Overall Assessment

  The theory is sound and the math is real. The field functional, decoherence criterion,
   tri-state collapse, gravity scheduling, and sheaf obstruction classification are all
  correctly implemented and empirically validated on live targets. Paper 4's claims
  about what the model does are defensible.

  The operational system is a proposal generator, not an engagement engine. The feedback
   loop from execution to evidence to state is broken at the session boundary. Every
  engagement after "proposal executed" requires a human to carry the evidence across the
   gap. The EternalBlue path realization (the paper's main empirical claim) was valid —
  but the Windows Server 2022 engagement that followed exposed six independent failures
  in the close loop.

  What this means for the paper: The formal contributions (field functional, fiber
  gravity, decoherence criterion, wicket graph) are solid and publishable. The empirical
   validation section accurately describes what was actually observed. The gap is in the
   "autonomous full-cycle" framing — that claim currently requires either: (a) fixing
  the P0 session detection items above, or (b) scoping the paper's claims precisely to
  "autonomous path selection and proposal generation" rather than "autonomous full
  engagement."

  The P0 fixes are targeted code changes — not architectural rewrites. The core
  architecture that makes SKG interesting is all there.
